# Scenario-1 Dashboard UI

This is the dedicated dashboard for the Scenario-1 Insider Threat Detection System. It visualizes alerts generated by the detection pipeline, applies ensemble logic to determine escalation, and provides a drill-down view for analysts.

## 1. System Architecture

The system consists of three main components:

1.  **Data Source**: `out/mvp0/test_run/alerts.ndjson`. This file is generated by the `src/run_loop.py` pipeline and contains raw detector outputs (Rules, ML, Anomaly).
2.  **Backend (FastAPI)**: `src/ui/app.py`.
    *   Loads the raw alerts.
    *   **Ensemble Logic**: Applies the specific Scenario-1 logic to combine Rule hits and ML scores into a final `ensemble_score` and `escalated` status.
    *   **State Management**: Handles the "Time Loop" functionality.
        *   Starts at the earliest day found in alerts.
        *   Automatically advances the simulated day by 1 every second (configurable).
        *   Pausing stops the time progression.
    *   **API**: Exposes REST endpoints for the frontend.
3.  **Frontend (HTML/JS/CSS)**: `src/ui/static/`.
    *   A single-page application that consumes the API.
    *   Renders the Dashboard, Heartbeat Graph, and Alert Details Modal.
    *   Updates in real-time as the backend simulation advances.

## 3. API Endpoints

| Endpoint | Method | Description |
| :--- | :--- | :--- |
| `/api/heartbeat?days=7` | GET | Returns daily alert counts (total and escalated) for the heartbeat graph. Respects `current_day`. |
| `/api/users?tab={all|escalated}` | GET | Returns a list of users with alerts up to `current_day`. |
| `/api/users/{user_key}/alerts` | GET | Returns all visible alerts for a specific user, sorted by day descending. |
| `/api/alerts/{alert_id}` | GET | Returns the full alert object, including the `ensemble_explanation`. |
| `/api/state` | GET | Returns current system state (`paused`, `current_day`, `earliest_day`, `latest_day`). |
| `/api/state/pause` | POST | Pauses/Unpauses the system. Body: `{"paused": true}`. Stops time progression. |

## 4. Ensemble Logic

The ensemble logic is defined in `src/ui/app.py` -> `compute_ensemble`. It combines inputs as follows:

1.  **Post-Departure Chain** (`s1_chain_post_departure`): **Score 1.0 (Escalated)**.
2.  **Standard Chain** (`s1_chain`): **Max(0.9, ML Score)**. Escalated if >= 0.9.
3.  **Near Miss** (`s1_near_miss`): **Max(0.5, ML Score * 0.75)**. Escalated if >= 0.9.
4.  **No Rules**: **ML Score**. Escalated if >= 0.9.

*Note: Anomaly scores are currently placeholders and do not influence the ensemble.*

## 5. Extension Guide

### Wiring Anomaly Detector
1.  Update `src/ui/app.py`: Modify `compute_ensemble` to incorporate `anomaly_score`.
2.  Update `src/ui/static/app.js`: Update `openModal` to display the active anomaly score and contribution.

### Adding Forecasting
1.  Ensure forecast scores are present in `alerts.ndjson`.
2.  Update `normalize_alert` in `src/ui/app.py` to extract `forecast_score`.
3.  Update the UI to display this score (e.g., add a new card in the modal).

### Modifying Rules
Rule definitions are in `src/detector/rules.py`. If you add new rules, update `compute_ensemble` in `src/ui/app.py` to handle the new rule IDs (e.g., assign weights or override logic).
